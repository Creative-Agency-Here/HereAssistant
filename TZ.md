# HereAssistant — ТЗ для следующего агента

> Это ТЗ передаётся в новый тред (внутри самого HereAssistant) и используется как «семя» для дальнейшей разработки. Агент работает над своим же кодом, читает этот файл, выполняет задачи по приоритету, обновляет хронологию в конце документа после каждой итерации.

---

## 0. Контекст проекта

**HereAssistant** — личный Telegram-бот, выступающий мостом к нескольким CLI-ассистентам (Claude Code, Codex, Gemini). Один админ (владелец), несколько изолированных аккаунтов на каждый провайдер, переключение через команды в Telegram.

**Текущая локация:** `C:\Users\Administrator\Desktop\HereAssistant\` (Windows Server 2022).

**Структура проекта на момент передачи ТЗ:**

```
HereAssistant/
├── bot.py              # сам бот: aiogram + SQLite + 3 CLI-провайдера
├── manage.py           # CLI-меню для управления аккаунтами/деплоя
├── requirements.txt    # aiogram
├── README.md           # документация
├── .env                # TELEGRAM_BOT_TOKEN, ADMIN_TELEGRAM_ID, CLAIM_CODE
├── bridge.sqlite3      # вся БД
└── .runtime/
    ├── cli_homes/      # изолированные папки аккаунтов
    └── logs/bot.log    # текущий лог
```

**Архитектурные решения, которые трогать нельзя:**
- Изоляция CLI-аккаунтов через подмену env-переменных (`CLAUDE_CONFIG_DIR`, `CODEX_HOME`, `USERPROFILE`+`HOME`)
- На Windows запуск CLI через `cmd /c <полный путь к .cmd>` (npm-обёртки)
- SQLite как единое хранилище истории, аккаунтов, разговоров
- Per-thread `asyncio.Lock` для последовательной обработки в одном чате
- Claim-механика: пустой `ADMIN_TELEGRAM_ID` → бот печатает claim-ссылку → первый, кто перешёл с правильным кодом → админ

## 1. Задачи к реализации (по приоритету)

### 1.1. РАСШИРЕННОЕ ЛОГИРОВАНИЕ С РОТАЦИЕЙ ПО ДНЯМ

**Что не так сейчас:** один файл `.runtime/logs/bot.log`, всё пишется в кучу.

**Сделать:**

- Логи писать в `.runtime/logs/bot-YYYY-MM-DD.log` — новый файл каждый день в полночь по локальному времени сервера (`logging.handlers.TimedRotatingFileHandler`, `when='midnight'`).
- Хранить логи за последние **30 дней**, старше — автоудалять (`backupCount=30`).
- Дополнительно — **структурированный лог событий** в SQLite (новая таблица `events`):

```sql
CREATE TABLE IF NOT EXISTS events (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp       INTEGER NOT NULL,
    event_type      TEXT NOT NULL,   -- 'message_in' | 'message_out' | 'tool_call' | 'error' | 'deploy' | 'reload' | 'switch_account' | 'switch_model'
    user_id         INTEGER,
    chat_id         INTEGER,
    thread_id       INTEGER,
    account_label   TEXT,
    provider        TEXT,
    model           TEXT,
    tokens_in       INTEGER,
    tokens_out      INTEGER,
    duration_ms     INTEGER,
    payload         TEXT             -- JSON с деталями
);
CREATE INDEX IF NOT EXISTS idx_events_ts ON events(timestamp);
CREATE INDEX IF NOT EXISTS idx_events_type ON events(event_type, timestamp);
```

**В каждом событии писать минимум:**
- кто (`user_id`, username)
- куда (`chat_id`, `thread_id`)
- какой аккаунт/провайдер/модель
- сколько токенов (если CLI это отдаёт — парсить из stdout)
- сколько мс заняло
- содержание (для `message_in`/`message_out` — первые 500 символов + общая длина)

### 1.2. КОМАНДЫ СТАТИСТИКИ В TELEGRAM

Добавить:

- `/stats` — за последние 24 часа: сколько сообщений, к каким моделям, суммарно токенов in/out, среднее время ответа
- `/stats week` — то же за 7 дней с разбивкой по дням
- `/stats all` — общая статистика по всем аккаунтам с тех пор, как ведётся учёт
- `/log` — последние 20 событий из таблицы `events` человекочитаемо
- `/log error` — только ошибки за последние 24 часа

Формат — компактные таблицы или списки, чтобы помещалось в одно сообщение Telegram (4096 символов).

### 1.3. РУССКИЙ ЯЗЫК У АГЕНТОВ

Сейчас все три CLI отвечают по-английски по умолчанию.

**Сделать:**

- Перед каждым запросом к провайдеру добавлять системную инструкцию: *«Отвечай на русском языке. Используй русский для пояснений, комментариев в коде и сообщений пользователю. Технические термины и имена переменных оставляй на английском.»*
- Для Claude Code — через `--system-prompt` или дописыванием в начало промпта
- Для Codex — через `model_instructions_file` (создать `bridge_agent_instructions.md`)
- Для Gemini — в начало промпта

Реализация в `bot.py`:

```python
RU_SYSTEM_INSTRUCTION = (

    "Отвечай на русском языке. "

```

И прокидывать в каждый `provider.run()`.

### 1.4. УБРАТЬ MARKDOWN, КОТОРЫЙ НЕ РЕНДЕРИТСЯ

CLI часто отвечают с markdown (`**жирный**`, `### заголовки`, `` ` `` инлайны). Telegram умеет MarkdownV2, но он требует экранирования спецсимволов, иначе ломается.

**Сделать:**

- Реализовать функцию `to_telegram_md(text)`, которая:
  - либо корректно экранирует MarkdownV2 (`_*[]()~`>#+-=|{}.!`)
  - либо чистит markdown до plain text (убирает `**`, `###`, оставляет code blocks как есть)
- Сделать **переключатель в `.env`**: `TELEGRAM_MARKDOWN_MODE=clean|escaped|off`. Дефолт — `clean` (plain text).
- При отправке `message.answer(text, parse_mode=...)` использовать соответствующий режим.

### 1.5. INLINE-КНОПКИ ВМЕСТО ТЕКСТОВЫХ ОТВЕТОВ

Когда бот задаёт пользователю выбор (например, «продолжить?», «какой аккаунт?»), сейчас приходится писать текстом.

**Сделать:**

- Использовать `InlineKeyboardMarkup` от aiogram там, где это уместно
- Команда `/account` без аргументов — показать список аккаунтов кнопками, нажатие → переключение
- Команда `/model` без аргументов — кнопки с популярными моделями текущего провайдера
- Подтверждения вроде `/reset` — кнопка «Да, очистить» / «Отмена»
- `/cwd` без аргументов — показать недавние папки + кнопки

### 1.6. КОМАНДА `/help` И АВТОПОДСКАЗКА `/`

В Telegram если набрать `/` — должен появиться список команд с описанием. Это настраивается через `setMyCommands` у Bot API.

**Сделать:**

- При старте бота вызывать `bot.set_my_commands([BotCommand(command="...", description="...")])`
- Команда `/help` — длинный список с подробностями по каждой команде (с примерами)

### 1.7. РАБОЧАЯ ПАПКА ПО УМОЛЧАНИЮ — ВНУТРИ ПРОЕКТА

Сейчас `DEFAULT_CWD=C:\Users\Administrator` — небезопасно, потому что Claude может случайно нагадить в системные файлы.

**Сделать:**

- При первом запуске создавать папку `workspace/` рядом с `bot.py`:
  ```
  HereAssistant/
      workspace/
          default/    # cwd по умолчанию для новых тредов
  ```
- `DEFAULT_CWD` в `.env` менять на `${BASE_DIR}/workspace/default`
- Для каждого нового треда — опционально создавать подпапку `workspace/thread-<chat_id>-<thread_id>/`
- В Telegram добавить команду `/project new <name>` — создаёт `workspace/<name>/` и устанавливает её как `cwd`
- `/project list` — показать существующие проекты-папки в `workspace/`
- `/project use <name>` — переключить cwd на эту папку

### 1.8. САМОПЕРЕЗАПУСК (HOT-RELOAD)

Сейчас, чтобы применить изменения в `bot.py`, надо вручную остановить и запустить.

**Сделать команду `/deploy` в Telegram:**

Логика:
1. Бот пишет: «🔄 Перезапускаюсь, ждите 5-10 секунд...»
2. Бот сохраняет в `.runtime/state/restart.json`:
   - `timestamp_before` — когда стартовал перезапуск
   - `chat_id`, `thread_id`, `message_id` — куда написать после старта
   - `version_before` — текущий hash файла `bot.py` (SHA256 от содержимого)
3. Бот вызывает `os.execv(sys.executable, [sys.executable, sys.argv[0]])` — заменяет текущий процесс новым Python с тем же `bot.py`
4. При старте `bot.py` проверяет `.runtime/state/restart.json`:
   - если есть — читает, считает `duration = now - timestamp_before`
   - вычисляет новый hash → diff с предыдущим (число изменённых строк через `difflib`)
   - удаляет `restart.json`
   - отправляет в `chat_id` сообщение:
     ```
     ✓ Перезапуск завершён за 7.3 сек
     Изменения: +12 -3 строк в bot.py
     Версия: 4f2a1c → b8e7d2
     ```
5. Если перезапуск был не через `/deploy`, а обычный — никакого сообщения не слать (просто `Bot started`).

Дополнительно — команда `/version` показывает текущий SHA256 короткий + дату последнего изменения `bot.py`.

После каждой итерации (когда агент закончил блок работы и собирается деплоить) — **обязательно** дополнить раздел «Хронология» в конце этого ТЗ:

Это нужно, чтобы следующий агент (или ты же в новой сессии) видел картину развития и не повторял работу.

## 2. Правила работы для агента

1. **Не ломай существующее.** Сначала тестируй на маленьком фиксе, потом дeplой.
2. **Один деплой = одна логическая фича.** Не накапливай 5 фич в одной правке.
3. **Перед деплоем** покажи владельцу краткий diff (через `/diff` придумай команду, или просто текстом в чат) — что именно меняется.
4. **После деплоя** — отчёт: что добавлено, что сломалось (если что), что осталось.
5. **Каждое изменение `bot.py`** — обязательная проверка `python -m py_compile bot.py` локально перед `/deploy`.
6. **`.env`, `bridge.sqlite3`, `.runtime/cli_homes/`** — никогда не трогай и не удаляй.
7. **Резервная копия** — перед каждым `/deploy` копировать текущий `bot.py` в `.runtime/backups/bot-YYYY-MM-DD-HHMM.py` (хранить последние 20).
8. **Не выводи токены в логи.** Маскируй `TELEGRAM_BOT_TOKEN`, OAuth-токены, `claude.credentials.json`.
9. **Документация и README обновляются** одновременно с фичей, не «потом».
10. **Если фича требует новой колонки в SQLite** — миграция через `ALTER TABLE` с проверкой существования.

## 3. Чего НЕ делать

- Не подключать LangChain, не подключать LiteLLM, не делать «универсальный роутер моделей через API-ключи» — у нас принципиально CLI-подписки, не API.
- Не пытаться обходить лимиты ротацией нескольких аккаунтов одного провайдера в авто-режиме (`/account use <label>` вручную — да; крон, который сам переключает каждые N минут — нет).
- Не делать веб-интерфейс/dashboard в первой итерации. Telegram достаточно.
- Не подключать второго пользователя/мультитенантность. Бот для одного админа.
- Не трогать структуру изоляции CLI-аккаунтов (это рабочее, проверенное решение).

**Известные ограничения (для следующего агента):**

- Парсинг токенов из ответа CLI пока работает только для Claude Code (поле `usage` в JSON). Codex и Gemini в `meta` возвращают пустой dict — `tokens_in`/`tokens_out` будут NULL в `events`. Можно улучшить, если CLI это выдают.
- Парсинг `session_id` из Codex — эвристика по строкам в stdout/stderr. Может не работать на новых версиях. Если ломается — смотреть формат вывода и править регулярку в `providers/codex.py`.

### 2026-05-24 — Итерация 0.10: permission mode + стриминг + диагностика

**Контекст бага:**
Claude Code в `--print` режиме по умолчанию работает с подтверждением прав на запись («Запросил разрешение на запись в providers/base.py — но оно не одобрено»). В TUI пользователь жмёт «yes», но через CLI подтверждать некому, поэтому ассистент зависает и отвечает текстом «дай мне права».

Параллельно — пользователь попросил, чтобы Telegram-сообщение **обновлялось в процессе**, а не висело пустым 60-180 секунд, пока CLI генерирует ответ.

**Что сделано:**

1. **`CLAUDE_PERMISSION_MODE` в `.env`** (дефолт `acceptEdits`) — позволяет правки без подтверждений. Передаётся в `claude --permission-mode <value>`.
2. **Стриминг ответа Claude через `--output-format stream-json`**:
   - `claude_code.py` теперь читает stdout построчно и парсит JSON-события (`system/init`, `assistant`, `tool_use`, `result`)
   - Каждый delta-чанк аккумулируется и передаётся в `progress_cb`
   - Заодно ловим `tool_use` события — пользователь видит `🔧 использует инструмент: <name>`
3. **`handlers/messages.py`** — live-обновление Telegram-сообщения:
   - При первом куске прогресса бот **создаёт сообщение** с маркером `⌛` и текстом
   - Каждые `PROGRESS_MIN_INTERVAL_SEC` секунд (дефолт 1.5) — `edit_text` с новым накопленным текстом
   - При финале — последний edit без `⌛` (или дополнительные сообщения если ответ > 3950 символов)
   - Защита от `TelegramBadRequest "message is not modified"`
4. **Новые ENV-переменные** в `manage.py`:
   - `CLAUDE_PERMISSION_MODE=acceptEdits`
   - `PROGRESS_ENABLED=1`
   - `PROGRESS_MIN_INTERVAL_SEC=1.5`
   - `TELEGRAM_MARKDOWN_MODE=clean`
5. **Сигнатура `provider.run()`** — добавлен опциональный параметр `progress: ProgressCb`. Codex и Gemini принимают его, но пока игнорируют (стрим у них устроен иначе — следующая итерация).
6. **Логирование** в `_exec`:
   - перед запуском — `prompt_preview` (первые 200 символов аргумента-промпта)
   - после — `rc`, `out_chars`, `err_chars`
   - в `messages.py` — `text=...` лог при приёме

**Файлы:**
- `providers/claude_code.py`: полностью переписан под streaming (~135 строк)
- `providers/base.py`: убран `system_preamble`, добавлено логирование, `progress=None` в сигнатуре
- `providers/codex.py`, `providers/gemini.py`: параметр `progress=None` (no-op)
- `handlers/messages.py`: live-обновление через `progress_cb`, обработка переполнения
- `manage.py`: расширенный `env_template`

**Известные ограничения:**

- Стриминг работает только для Claude Code. Для Codex стрима нет — он отдаёт всё одним блоком в `codex exec`. Чтобы прикрутить — нужно переключиться на `codex exec --json` и парсить SSE-подобные события. Следующая итерация.
- Для Gemini в `-p` режиме стрима нет в принципе. Можно посмотреть `gemini chat` с интерактивным stdin, но это сложнее.
- Telegram имеет жёсткий rate limit на `edit_text` (примерно 1 раз/сек на одно сообщение). `PROGRESS_MIN_INTERVAL_SEC=1.5` — безопасный дефолт, но если хочешь чаще — рискуешь получить `Too Many Requests`.
- При `--include-partial-messages` у Claude формат event-ов может слегка отличаться по версиям. Если стрим вдруг сломается — проверь сырой stdout через `claude --print --output-format stream-json --verbose --include-partial-messages "test"` и подправь парсинг event-ов в `_run_streaming`.
- Если ответ Claude > 3800 символов — live-апдейт обрывается, финал отправляется отдельным сообщением. Это потому что Telegram режет одно сообщение на 4096 символов, и редактировать «по кускам» неудобно.

**Что для следующей итерации:**
1. Протестировать на живом сервере (на момент написания не запускалось)
2. Прикрутить стрим к Codex через `--json` события
3. Возможно, прогресс-индикатор для долгих tool_use (например, при чтении большого файла Claude может думать минуту)
4. Кнопка «Прервать» в live-сообщении, которая киляет subprocess

**Версия после деплоя:** см. `/version`

### 2026-05-25 — Итерация 0.11: typing-heartbeat, прерывание, подпись, диагностика стрима

**Контекст:**

Логи v0.10 показали, что `--output-format stream-json` запускается успешно (`rc=0`, `out_chars=2000`), но **колбэк прогресса ни разу не вызвался** — парсер событий пропустил все типы. Скорее всего, формат событий отличается от ожиданий.

Параллельно пользователь попросил:
- модель/время/обновлено в подпись каждого ответа
- постоянный typing (он гаснет через 5 секунд, надо обновлять)
- прерывание текущего процесса новым сообщением (не очередь, а cancel)
- +N −M строк по итогам tool_use
- веб-UI как VS Code (отложено в следующую итерацию)

**Что сделано:**

1. **Парсер `claude_code.py` переписан** под несколько форматов:
   - `system/init` — захватывает session_id
   - `assistant` с `message.content[]` — эвристика «если новый текст содержит старый — заменяем, иначе конкатим»
   - `stream_event` с `content_block_delta` / `content_block_start` / `content_block_stop` — partial chunks
   - `tool_use` — отдельным событием **и** внутри `stream_event`
   - `result` — финал с `usage` / `session_id`
   - Все типы событий считаются в `events_seen`, итог в лог: `events={'system': 1, 'assistant': 5, ...}`

2. **`CLAUDE_DEBUG_STREAM=1`** — включает dump сырых событий в `.runtime/logs/claude-stream-<ts>.jsonl`. Если стрим не работает — включи, посмотри что приходит, доадаптируй парсер.

3. **Typing-heartbeat в `messages.py`** — фоновая задача обновляет `chat_action("typing")` каждые `TYPING_INTERVAL_SEC` (4 сек). Останавливается через `asyncio.Event`.

4. **Прерывание новой репликой** — словарь `_active_tasks[(chat, thread)] → Task`. Новое сообщение → `task.cancel()` + «⏸ Предыдущий запрос прерван». Лок убран, теперь не очередь, а вытесняющая отмена. Управляется `INTERRUPT_ON_NEW_MESSAGE=1`.

5. **Подпись `_format_signature`** в конце ответа:
   ```
   — claude-opus-4-7 · 24.3с · +12 −3 строк · 1 файл · обновлено 04:55:12
   ```

6. **Подсчёт правок** — провайдер парсит `tool_use` `Edit`/`Write`/`MultiEdit`, извлекает `old_string`/`new_string`/`file_path`, считает строки по `\n`. Складывает в `meta["edits"]`.

7. **Live-сообщение с текущим tool**:
   ```
   ⌛ 17с · 🔧 Edit
   
   Сейчас правлю providers/base.py, добавляю логирование...
   ```

8. **Новые ENV**: `CLAUDE_DEBUG_STREAM=0`, `TYPING_INTERVAL_SEC=4`, `INTERRUPT_ON_NEW_MESSAGE=1`.

**Файлы:**
- `providers/claude_code.py`: широкий парсер + dump
- `handlers/messages.py`: typing-heartbeat, _active_tasks, подпись, защита от cancel
- `providers/codex.py`, `providers/gemini.py`: only `progress=None` signature
- `manage.py`: расширенный `env_template`

**Известные ограничения:**
- Если у Claude совсем другой формат событий — включить `CLAUDE_DEBUG_STREAM=1`, посмотреть `claude-stream-*.jsonl`, доадаптировать парсер
- При `task.cancel()` subprocess Claude не убивается явно — остаётся висеть. Следующая итерация: явный `proc.kill()` в обработчике `CancelledError`
- INTERRUPT при частых сообщениях может превысить Telegram rate limit
- Подсчёт строк — эвристика по `\n`

**Что для следующей итерации:**

1. **Веб-UI** — см. ниже отдельным блоком
2. **Kill subprocess при cancel** — `proc.kill()` в `claude_code.py` обработчике
3. **Стрим для Codex** через `codex exec --json`
4. **Команда `/diff`** после ответа — полный diff правок

**Версия:** см. `/version`
